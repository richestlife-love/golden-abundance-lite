# Phase 4 — Wire Frontend to Backend

**Status:** Design (approved)
**Date:** 2026-04-20
**Predecessors:** [Phase 2 API contract](./2026-04-19-phase-2-api-contract-design.md), [Phase 3 routing](./2026-04-20-phase-3-routing-design.md)
**Successor (planning):** Phase 6 (real Google OAuth + refresh tokens)

## 1. Goal

Replace the frontend's in-memory mock state and demo seeds with live data fetched from the FastAPI backend completed in Phase 5. After this phase, every screen renders from the TanStack Query cache; `AppStateContext` is deleted; the only client-only state is transient UI overlays (success modal, toasts).

This phase ships behind the existing `/auth/google` stub — Phase 6 swaps the stub for real Google OAuth without touching anything described here except `auth/session.ts`'s `signIn` body.

## 2. Decisions (recap from brainstorm)

| # | Decision | Choice |
|---|---|---|
| 1 | State source of truth | TanStack Query cache; gut `AppStateContext`, keep tiny `UIStateProvider` for ephemeral UI state |
| 2 | Auth flow | Real `POST /auth/google` against the existing stub; JWT in `localStorage`; Bearer header on every request |
| 3 | API client shape | `openapi-typescript` for schema types + hand-rolled `apiClient` + per-resource modules |
| 4 | Mutations | Invalidate-on-success default; optimistic only for join-request approve/reject and team rename/topic patch |
| 5 | Routing/loading | TanStack Router loaders call `ensureQueryData`; components `useSuspenseQuery`; route-level `pendingComponent` + `errorComponent` |
| 6 | Demo data | Move all to backend seed; pending-request fan-out uses γ-with-two-leaders pattern (jet + ami both demo leaders) |
| 7 | Tests | MSW for fetch interception; real QueryClient per test |
| 8 | 401 handling | Global interceptor → clear token + `queryClient.clear()` → redirect to `/sign-in?returnTo=...` with toast |

Three additional cross-cutting conventions — field naming, resource URLs, and 401-handler wiring — were not resolved during brainstorming. They are defined in §3.4.

## 3. Architecture

### 3.1 Module layout

```
    client.ts            # apiFetch wrapper, bearer injection, 401 interceptor
    errors.ts            # ApiError class
    auth.ts              # postGoogleAuth, postLogout
    me.ts                # getMe, getMyTasks, getMyTeams, getMyRewards, postProfile, patchMe
```

`AppStateContext.tsx` is deleted. Every `useAppState()` callsite migrates to the appropriate `useQuery` / `useMutation` / `useMe` / `useUIState` hook. No compatibility shim.

### 3.2 What survives client-side

`UIStateProvider` (~50 LOC):

- `successData` + `setSuccessData` — celebration modal overlay
- `toasts` + `dismissToast` — transient banners (mutation errors, 401 expiry message). The enqueue side (`pushToast`) is a module-level late-bound function in `ui/toasts.ts`, not a context hook — so `signOut` (module-scoped; see §4.4) and mutation `onError` callbacks can emit toasts from outside the React tree. Wiring in §3.4.

That's it. Everything else moves to the server cache or is derived at the call site.

### 3.3 What gets deleted

From the current `AppStateContext`:

- `tasks`, `ledTeam`, `joinedTeam` and every handler that mutates them
- `user`, `profileComplete` — replaced by `useMe()`; `profile_complete` is read directly from the server's `User.profile_complete` (authoritative — set atomically by `POST /me/profile`, see Phase 2 §1.2), never client-derived from `zh_name`
- `userIdFromEmail` — server issues UUIDs; never derived client-side again
- `syncTeamTask` — task-3 progress is computed server-side and returned in `GET /me/tasks` (already true per Phase 5d)
- `simulateJoinApproved` — replaced by a real `POST .../approve` call in the seeded demo flow
- The hardcoded `林詠瑜 / 陳志豪 / 王美玲` request seed in `handleProfileComplete` — moves to backend seed

From `frontend/src/data.ts`: the `TASKS` mock array. Type re-exports stay if anything imports them.

From `frontend/src/screens/RankScreen.tsx`: the ~43 KB of inline mock leaderboard data. Live data comes from `/rank/*`; any small fixtures the tests need move to `src/test/msw/fixtures.ts`.

### 3.4 Cross-cutting conventions

Three conventions cut across every section below. Pinning them here so downstream sections can reference them without relitigating.

**Field naming — snake_case on the client.** Phase 2 §6 deferred this. The OpenAPI schema (and therefore `schema.d.ts`) is snake_case; rather than shim a transform layer into `apiFetch` and maintain two parallel naming worlds, every client identifier becomes snake_case to match: `User.zh_name`, `Task.est_minutes`, `Team.week_points`, `UserRef.avatar_url`, `TeamChallengeProgress.joined_total`, and so on. `frontend/src/types.ts` is deleted entirely; each `api/*.ts` re-exports its named aliases from `schema.d.ts` and components import from there. The rename is mechanical and happens one screen at a time during PR 2 (converting a screen is "swap `useAppState()` for `useSuspenseQuery` AND rename camelCase fields to snake_case in the same commit"). No transform layer, no compatibility shim, no camelCase survivors.

This affects roughly 160 call-sites across the frontend (see e.g. `types.ts:46-56`, `AppStateContext.tsx:96-133`); scope is contained because PR 2 already touches every read-side screen.

**Resource URLs — `display_id` in the path.** URLs use the human-readable short code: `/tasks/T3`, not `/tasks/<uuid>`. The contract's `display_id` regex (`T[0-9A-Z]+`, `U[A-Z0-9]{3,7}`, `T-[A-Z0-9]{3,10}`) can never collide with a UUID, so there's no ambiguity. The backend's `GET /tasks/{task_id}` is UUID-typed; the frontend resolves `display_id → UUID` against the cached `myTasks` list before calling it:

```ts
// routes/_authed.tasks.$taskId.tsx
loader: async ({ params, context }) => {
  await context.queryClient.ensureQueryData(myTasksQueryOptions());
  const list = context.queryClient.getQueryData<Task[]>(qk.myTasks) ?? [];
  const task = list.find((t) => t.display_id === params.taskId);
  if (!task) throw notFound();
  return context.queryClient.ensureQueryData(taskQueryOptions(task.id));
},
```

The `_authed` parent loader warms `myTasks` (§4.3), so the resolution works on cold loads too. No team-detail route exists in Phase 3, so teams don't need solving here — `GET /teams/{team_id}` stays UUID-only until a team-detail deep link ships.

Internal navigations pass `display_id`: `navigate({ to: '/tasks/$taskId', params: { taskId: 'T3' } })`. The old hardcoded `"1" | "2" | "3"` in `routes/_authed.tasks.$taskId.start.tsx:8` and `screens/MyScreen.tsx:60,755` becomes `"T1" | "T2" | "T3"`; the form-dispatch there switches from `id === 1 → InterestForm` to `task.form_type === 'interest' → InterestForm` (source of truth is the task's own `form_type`, not a URL heuristic).

**Late-bound handlers — 401 interceptor and toast emission.** Two hooks bridge module-scoped callers (the 401 interceptor in `api/client.ts`; the module-scoped `signOut` in `auth/session.ts`) to state that lives in the React tree. `api/client.ts` must not import `auth/session.ts` (the session module depends on `api/auth.ts` for `postLogout`, creating a cycle), and `signOut` — which runs outside any provider — cannot reach `UIStateProvider`'s state directly. Both use the same registrar pattern: a module-level setter overrides a null default, callers invoke through a null-safe wrapper.

**401 handler:**

```ts
// api/client.ts
let onSessionExpired: ((opts: { returnTo: string }) => void) | null = null;
export function setSessionExpiredHandler(fn: typeof onSessionExpired): void {
  onSessionExpired = fn;
}
```

```ts
// auth/session.ts — top-level side effect, runs at module init
import { setSessionExpiredHandler } from '../api/client';
setSessionExpiredHandler(({ returnTo }) => signOut({ reason: 'expired', returnTo }));
```

**Toast handler:**

```ts
// ui/toasts.ts
export type Toast = { kind: 'info' | 'error' | 'success'; message: string };
let sink: ((t: Toast) => void) | null = null;
export function setToastSink(fn: typeof sink): void { sink = fn; }
export function pushToast(t: Toast): void { sink?.(t); }
```

```ts
// ui/UIStateProvider.tsx — inside the provider component
useEffect(() => {
  setToastSink(enqueue);
  return () => setToastSink(null);
}, [enqueue]);
```

All callers (`signOut`, mutation `onError` defaults, screens) import `pushToast` from `ui/toasts.ts`. A null sink is a graceful no-op, so a toast fired before the provider mounts is dropped silently rather than crashing.

`auth/session.ts` is imported from `main.tsx` before the router mounts, so the session-expired handler is registered before any loader fires. `UIStateProvider` wraps the router, so its mount `useEffect` runs before any child-component query can 401. Tests wire their own handlers (or none) inside `renderRoute` (§8.2).

## 4. Auth lifecycle

### 4.1 Sign-in flow

1. User lands on `/sign-in` → `GoogleAuthScreen` reads `src/dev/demo-accounts.json` (generated from backend seed) and renders one button per demo account.
2. User picks an account → screen calls `signIn(email)` from `AuthProvider`.
3. `signIn` calls `postGoogleAuth({ id_token: email })`.
4. Backend stub validates → returns `AuthResponse`. Frontend:
   - `tokenStore.set(access_token)`
   - `queryClient.setQueryData(qk.me, user)` (seed the cache so `/home` doesn't refetch immediately)
   - If `search.returnTo` is present, `router.navigate({ to: returnTo })`; else navigate to `/home` (or `/welcome` if `!user.profile_complete`).

The Phase 5 stub (`backend/src/backend/auth/google_stub.py`) validates the `id_token` as an email-shaped string (via `email_validator`) and returns it lowercased as the authenticated identity. The frontend passes the email verbatim — there is no client-side encoding helper. Phase 6 swaps the stub for real Google ID-token verification; the only change to the frontend is `signIn`'s body (GIS popup → real `id_token` → `postGoogleAuth`), which is why `auth/session.ts` is the sole file Phase 6 touches in this layer.

### 4.2 Token storage

```ts
// auth/token.ts
const KEY = 'ga.token';
export const tokenStore = {
  get: () => localStorage.getItem(KEY),
  set: (t: string) => localStorage.setItem(KEY, t),
  clear: () => localStorage.removeItem(KEY),
};
```

No reactive wrapping. `apiFetch` reads `tokenStore.get()` synchronously per request. The `AuthProvider`'s `isSignedIn` boolean is mirrored React state (toggled by `signIn`/`signOut`) so route guards re-render on auth changes.

`AuthProvider` accepts an `initialToken` prop for tests. On mount it writes `initialToken` to `tokenStore` (otherwise `apiFetch` would read from an empty `localStorage` and every mocked request would 401 despite tests having "signed in" the provider). Production never passes `initialToken`.

### 4.3 Route guards

The `_authed` layout route owns both authed checks (token present + profile complete) AND warms the queries every authed child depends on:

```ts
// routes/_authed.tsx
beforeLoad: async ({ context, location }) => {
  if (!tokenStore.get()) {
    throw redirect({ to: '/sign-in', search: { returnTo: location.href } });
  }
  const me = await context.queryClient.ensureQueryData(meQueryOptions());
  if (!me.profile_complete) throw redirect({ to: '/welcome' });
},
loader: async ({ context }) => {
  // Warmed here so child loaders (task detail resolving display_id → UUID) can
  // read it synchronously from the cache. Keeps per-route loaders thin.
  await context.queryClient.ensureQueryData(myTasksQueryOptions());
},
```

`profile_complete` is server-authoritative (Phase 2 §1.2). No client derivation from `zh_name`.

`/sign-in` keeps its already-authed guard — a signed-in user typing `/sign-in` is bounced to wherever they should be:

```ts
// routes/sign-in.tsx
beforeLoad: async ({ context }) => {
  if (!tokenStore.get()) return;
  const me = await context.queryClient.ensureQueryData(meQueryOptions());
  throw redirect({ to: me.profile_complete ? '/home' : '/welcome' });
},
```

`/welcome` mirrors: no token → `/sign-in?returnTo=/welcome`; token + `profile_complete` → `/home`; otherwise stay.

### 4.4 401 interceptor

`api/client.ts` calls the late-bound handler (see §3.4 for why it's not a direct import):

```ts
if (res.status === 401) {
  onSessionExpired?.({
    returnTo: window.location.pathname + window.location.search,
  });
  throw new ApiError(401, 'Session expired');
}
```

`signOut({ reason, returnTo })` — registered as the handler, also called on user-initiated sign-out:

```ts
let inFlight = false;
async function signOut(opts: { reason?: 'expired' | 'user'; returnTo?: string } = {}) {
  if (inFlight) return;               // (1) burst-dedup: a parallel-query 401 storm calls us N times
  inFlight = true;
  try {
    // (2) Logout POST with the token still set. Fire-and-forget; for
    //     reason==='expired' the server 401s back, which is harmless — we
    //     use this for observability, not state change. Sent BEFORE clear()
    //     so user-initiated sign-outs carry a valid Bearer.
    const token = tokenStore.get();
    if (token) {
      void fetch('/api/v1/auth/logout', {
        method: 'POST',
        headers: { Authorization: `Bearer ${token}` },
      }).catch(() => {});
    }
    tokenStore.clear();                // (3) drop the token
    if (opts.reason === 'expired') {
      pushToast({ kind: 'info', message: '您的工作階段已過期，請重新登入' });
    }
    await router.navigate({            // (4) navigate first — clearing cache
      to: '/sign-in',                  //     before the authed subtree unmounts
      search: opts.returnTo ? { returnTo: opts.returnTo } : {},
    });
    queryClient.clear();               // (5) then drop caches; /sign-in has no authed queries
  } finally {
    inFlight = false;
  }
}
```

After successful sign-in, `signIn` reads `search.returnTo` from the router state and navigates there instead of `/home`.

### 4.5 Token TTL

Keep whatever the backend currently mints (`jwt_ttl_seconds` defaults to 86400 in `backend/src/backend/config.py`). A dev never hits natural expiry in one session, so don't rely on it to exercise the interceptor — the dedicated 401-interceptor test in §8.4 (MSW handler returns 401 → assert redirect + toast) is the coverage.

## 5. Data layer

### 5.1 Type generation

New justfile recipe `gen-types`. It generates offline from FastAPI's in-process `app.openapi()` — no running server, no DB. CI just runs the recipe before typecheck.

```
gen-types:
    uv run --project backend python -c 'import json; from backend.server import app; print(json.dumps(app.openapi()))' > /tmp/ga-openapi.json
    pnpm -C frontend dlx openapi-typescript /tmp/ga-openapi.json -o src/api/schema.d.ts
```

Generated file is gitignored. Running the recipe requires `uv sync` in `backend/` (which CI does anyway for the backend build). Per-resource modules import via `import type { components } from './schema'` and re-export named aliases:

```ts
export type User = components['schemas']['User'];
export type Task = components['schemas']['Task'];
// ...
```

### 5.2 `apiFetch`

```ts
let onSessionExpired: ((opts: { returnTo: string }) => void) | null = null;
export function setSessionExpiredHandler(fn: typeof onSessionExpired): void {
  onSessionExpired = fn;
}

export async function apiFetch<T>(path: string, init?: RequestInit): Promise<T> {
  const token = tokenStore.get();
  const res = await fetch(`/api/v1${path}`, {
    ...init,
    headers: {
      'Content-Type': 'application/json',
      ...(token ? { Authorization: `Bearer ${token}` } : {}),
      ...init?.headers,
    },
  });
  if (res.status === 401) {
    onSessionExpired?.({ returnTo: window.location.pathname + window.location.search });
    throw new ApiError(401, 'Session expired');
  }
  if (!res.ok) {
    const detail = await res.json().catch(() => ({})).then(j => j.detail ?? res.statusText);
    throw new ApiError(res.status, detail);
  }
  if (res.status === 204) return undefined as T;
  return res.json();
}
```

`auth/session.ts` registers the handler at module import time; `main.tsx` imports `auth/session.ts` before mounting the router. See §3.4.

Vite dev server proxies `/api/*` → `http://localhost:8000` (config in `vite.config.ts`). Production uses same-origin or `VITE_API_BASE_URL` override.

### 5.4 `queryOptions` pattern

Each `queries/*.ts` exports factories usable by both loaders and components:

```ts
// queries/me.ts
export const meQueryOptions = () => queryOptions({
  queryKey: qk.me,
  queryFn: () => api.me.getMe(),
  staleTime: 60_000,
});
```

Route loader:

```ts
loader: ({ context }) => context.queryClient.ensureQueryData(meQueryOptions()),
```

Component:

```ts
const { data } = useSuspenseQuery(meQueryOptions());
```

For routes whose URL segment is a `display_id` (see §3.4), the loader first warms the parent list, then resolves to a UUID:

```ts
// routes/_authed.tasks.$taskId.tsx
loader: async ({ params, context }) => {
```

`router` is created with `context: { queryClient }` so loaders and `beforeLoad` can reach it.

## 6. Mutations

### 6.1 Invalidation map

| Mutation | Invalidates on success |
|---|---|
| `POST /me/profile` (complete) | `qk.me`, `qk.myTeams`, `qk.myTasks` |
| `PATCH /me` | `qk.me`, `qk.myTeams`, and the broad prefix `['teams']` (user's `name` is denormalized into every team's `leader`/`members` refs server-side, so any cached team or search result is stale) |
| `POST /tasks/:id/submit` | `qk.task(id)`, `qk.myTasks`, `qk.myRewards`, `qk.me`, `['rank']` |
| `POST /teams/:id/join-requests` | `qk.team(id)`, `qk.myTeams` |
| `DELETE /teams/:id/join-requests/:req` | `qk.team(id)`, `qk.myTeams` |
| `POST .../approve` (leader) | optimistic patch on `qk.myTeams` (§6.2); on settle `qk.myTeams`, `qk.team(id)` (defensive — no Phase 3/4 subscriber), `qk.myTasks`, `qk.myRewards`, `qk.me`, `['rank']` |
| `POST .../reject` (leader) | optimistic patch on `qk.myTeams` (§6.2); on settle `qk.myTeams`, `qk.team(id)` (defensive) |
| `POST /teams/:id/leave` | `qk.team(id)`, `qk.myTeams`, `qk.myTasks`, `qk.myRewards`, `qk.me`, `['rank']` |
| `PATCH /teams/:id` (rename/topic) | optimistic patch (§6.2); on settle `qk.team(id)`, `qk.myTeams`, `['teams']` (search results) |
| `POST /auth/logout` | `queryClient.clear()` |

The reward cascade in approve is a known backend N+1 (Phase 5c debt). The frontend just consumes whatever rewards the server returns.

### 6.2 Optimistic mutations (three only)

All three patch `qk.myTeams` — the `MeTeamsResponse.led` slot — because `MyScreen.tsx` (and `RenameTeamSheet`) renders `ledTeam` from `useMyTeams().data.led`, not from `qk.team(id)`. Phase 3/4 has no team-detail route, so `qk.team(id)` has no subscriber; writing optimistically to it would leave the visible UI stale until settle-time invalidation, defeating the point.

**Approve join-request:**

```ts
onMutate: async ({ teamId, reqId }) => {
  await queryClient.cancelQueries({ queryKey: qk.myTeams });
  const prev = queryClient.getQueryData<MeTeamsResponse>(qk.myTeams);
  const team = prev?.led;
  if (prev && team && team.id === teamId) {
    const req = team.requests?.find(r => r.id === reqId);
    if (req) {
      // `req.user` is already a complete UserRef ({id, display_id, name, avatar_url}) —
      // spread it whole so no required field is dropped.
      queryClient.setQueryData<MeTeamsResponse>(qk.myTeams, {
        ...prev,
        led: {
          ...team,
          members: [...team.members, req.user],
          requests: team.requests!.filter(r => r.id !== reqId),
        },
      });
    }
  }
  return { prev };
},
onError: (_err, _vars, ctx) => {
  if (ctx?.prev) queryClient.setQueryData(qk.myTeams, ctx.prev);
  pushToast({ kind: 'error', message: '審核失敗，請再試一次' });
},
onSettled: (_data, _err, { teamId }) => {
  queryClient.invalidateQueries({ queryKey: qk.myTeams });
  queryClient.invalidateQueries({ queryKey: qk.team(teamId) });
  queryClient.invalidateQueries({ queryKey: qk.myTasks });
  queryClient.invalidateQueries({ queryKey: qk.myRewards });
  queryClient.invalidateQueries({ queryKey: qk.me });
  queryClient.invalidateQueries({ queryKey: ['rank'] });
},
```

**Reject join-request:** same shape against `qk.myTeams.led`, `requests: requests.filter(...)`, no `members` push. Settle-time invalidates `qk.myTeams` (to reconcile) and `qk.team(id)` (defensive).

**Rename team / patch topic** (`PATCH /teams/:id`): optimistic string swap on `qk.myTeams.led` (patch the `name` / `alias` / `topic` field on the `led` team when `led.id === teamId`), rollback the whole `MeTeamsResponse` on error, invalidate `qk.myTeams`, `qk.team(id)` (defensive), and `['teams']` (search result `name` is denormalized) on settle.

### 6.3 Error surfacing

Every mutation hook accepts an optional `onError` callback. Default behavior: `pushToast` from `UIStateProvider`. 409s on `submit` and join-request creation get human-readable copy mapped from the response `detail`.

### 6.4 Success overlay trigger

The success modal (`FormSuccessOverlay` + `successData` on `UIStateProvider`) used to fire from handlers on `AppStateContext` (`completeTask`, `joinTeam`, `approveRequest`). It now fires from mutation `onSuccess` callbacks — the server's response carries the awarded `reward` (for `POST /tasks/{id}/submit` it's `TaskSubmissionResponse.reward`), so the client reads what to display from there rather than re-deriving from the task def:

```ts
// mutations/tasks.ts
useSubmitTask({
  onSuccess: ({ task, reward }) => {
    if (reward) pushSuccess({ color: task.color, points: task.points, bonus: reward.bonus });
  },
});
```

Same pattern for team-challenge completion: `useApproveJoinRequest.onSuccess` checks the returned `Team.members.length + 1 >= cap`, and if the `Team` crossed the cap, calls `pushSuccess` with task-3's copy. No server-derived `just_completed_tasks` field for now — the client inspects the cache after invalidation.

Six accounts. Each gets a profile-complete `UserRow` + an auto-created led `TeamRow` — call into the existing `services.user` and `services.team.create_led_team` functions, not duplicate logic.

**Pending join-request fan-out (γ-with-two-leaders, respecting Phase 5c at-most-one-pending invariant):**

The backend's `POST /teams/:id/join-requests` returns 409 if the caller already has a pending request to *any* team (Phase 5c). Each demo non-leader can therefore have at most one pending request total. To preserve a demo for both jet and ami, split the four non-leaders 2/2:

- `jet@demo.ga` and `ami@demo.ga` are both designated demo leaders.
- `alex` and `mei` have a pending request against **jet's** team (2 rows).
- `kai` and `yu` have a pending request against **ami's** team (2 rows).
- Total: 4 rows.

Outcome:

- Signing in as **jet** → 2 pending requests visible (alex, mei). Approve them → jet's team has 3 members; task 3 progress 3/6.
- Signing in as **ami** → 2 pending requests visible (kai, yu). Same pattern, independent scenario.
- Signing in as alex / mei / kai / yu → empty led team + 1 outgoing pending request visible on `MyScreen`.

Two demo paths for "approve a request" without violating the at-most-one-pending invariant. The seed deliberately **does not** drive task 3 to completion: with six demo accounts and the one-pending rule, reaching a 6-member team requires out-of-band juggling (ami rejects kai/yu → kai/yu re-request jet → ami requests jet → jet approves all) that makes for a bad demo script. The approve UX is already fully exercised at 3/6 progress, which is the goal. Anyone who wants the completion modal can walk that dance or locally extend `DEMO_USERS`.

**Seed task progress:** none. All demo users start with empty `TaskProgressRow` so T1 / T2 / T4 demo cleanly. T3 (team completion) progresses naturally as the leader approves requests.

## 8. Test infrastructure

### 8.1 MSW setup

```ts
// src/test/msw/server.ts
import { setupServer } from 'msw/node';
import { defaultHandlers } from './handlers';
export const server = setupServer(...defaultHandlers);
```

Vitest setup (`src/test/setup.ts` extension):

```ts
beforeAll(() => server.listen({ onUnhandledRequest: 'error' }));
afterEach(() => { server.resetHandlers(); queryClient.clear(); });
afterAll(() => server.close());
```

`onUnhandledRequest: 'error'` is the right strict default — any accidental live fetch surfaces as a test failure rather than a silent network call. If a third-party fetch ever needs to pass through (fonts, analytics), add an explicit pass-through handler; don't relax to `'warn'` globally.

Per-test override:

```ts
server.use(
  http.get('/api/v1/me', () => HttpResponse.json(fixtures.userJet)),
  http.post('/api/v1/tasks/:id/submit', () => HttpResponse.json(submitOk, { status: 200 })),
);
```

### 8.2 `renderRoute` helper

Updated wrapper:

```ts
<QueryClientProvider client={new QueryClient({ defaultOptions: { queries: { retry: false } } })}>
  <AuthProvider initialToken={opts.token ?? null}>
    <UIStateProvider>
      <RouterProvider router={router} />
    </UIStateProvider>
  </AuthProvider>
</QueryClientProvider>
```

A fresh `QueryClient` per test prevents cross-test cache bleed. `AuthProvider` writes `initialToken` to `tokenStore` on mount (see §4.2) — `apiFetch` reads from `tokenStore` synchronously, so tests that pass `opts.token = 'valid'` will have MSW handlers see a `Authorization: Bearer valid` header. Tests that omit `opts.token` simulate a signed-out caller.

The current `AppStateProvider`-wrapped helper and its `stateRef.current.handleSignIn(...)` escape hatch go away in PR 3; tests that need to seed authed state do so via `opts.token` plus `server.use(...)` handlers that return the expected `/me` payload.
